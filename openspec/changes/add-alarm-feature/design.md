## 背景

React App 目前沒有鬧鐘功能。需要新增鬧鐘功能，讓使用者可以設定、檢視、管理鬧鐘提醒。這是一個純前端功能，使用瀏覽器計時器觸發鬧鐘。

## 目標 / 非目標

**目標：**
- 提供鬧鐘設定介面（時間、標籤）
- 顯示鬧鐘列表
- 支援啟用/停用鬧鐘
- 支援刪除鬧鐘
- 鬧鐘觸發時顯示通知

**非目標：**
- 不實作鬧鐘音效上傳（使用系統音效或內建音效）
- 不實作鬧鐘重複模式（只支援單次鬧鐘）
- 不實作鬧鐘 snooze（延後功能）

## 決策

### 1. 鬧鐘資料儲存方式：localStorage
- **選擇**：使用 localStorage 持久化鬧鐘資料
- **理由**：純前端應用，無後端 API，localStorage 可在瀏覽器關閉後保留資料
- **替代方案**：僅使用 React state（缺點：頁面刷新後資料消失）

### 2. 鬧鐘觸發機制：真實 Date 區間比對與補償
- **選擇**：記錄「上一次檢查的 timestamp」，每秒檢查時比對「上一次時間」到「當前時間」的區間。為了避免跨日或長時間休眠漏洞，演算法會遍歷該區間所跨越的所有日期，對每天結合鬧鐘時分產生成 `Date` 物件，再確認此物件是否落在檢查區間內。同時監聽 `visibilitychange` 事件立即補償檢查。
- **理由**：若僅用相對分鐘數，當跨天或休眠時間大於 24 小時，計算將會出錯漏響。基於真實 Date 的區間比對最為穩健。

### 3. 音效與系統通知：HTML5 Audio + Web Notification API
- **選擇**：
  - 音效：使用 HTML5 `Audio` API 播放內建免版權音效，隨 `triggeredAlarms` 佇列的非空狀態來播放/停止。若自動播放受阻，在 Modal UI 顯示點擊提示以解除限制。
  - 通知：使用 `Notification` API。在頁面背景時發送，點擊通知引導 `window.focus()`。在 UI 提供「測試音效與授權」按鈕，以符合現代瀏覽器對於 Notification 與 Autoplay 必須由使用者互動觸發的規範。
- **理由**：純網頁 Modal 在背景無作用，桌面通知與音效必不可少。提供手動測試授權按鈕可完美避開瀏覽器權限限制。

### 4. 觸發與停用狀態同步：觸發即停用 + 佇列管理
- **選擇**：當時間區間符合時，**立即**在內部將鬧鐘停用（`disableAlarm`）並將其加入待顯示的 `triggeredAlarms` 佇列。關閉 Modal 僅清空佇列。
- **理由**：若在關閉 Modal 時才停用，使用者若在響鈴時重整頁面，該鬧鐘在 localStorage 仍為啟用但因時間已過再也不會響，形成殭屍鬧鐘。觸發即停用能防重複，且佇列能防止多個鬧鐘同時過期時 UI 被覆蓋漏響。

## 可測試性考量

- `useAlarms` hook 可獨立測試，驗證鬧鐘新增、刪除、切換功能
- localStorage 操作需 mock，避免影響測試環境
- 測試需覆蓋「時間跳躍補償」：模擬上一次檢查在 08:29，當前時間 08:31，鬧鐘 08:30 應被成功觸發。
- 測試需覆蓋「跨多天休眠補償」：例如上一次檢查在 6/7 23:50，目前在 6/9 00:10，期間內所有符合的鬧鐘都應被觸發。
- UI 元件使用 `@testing-library/react` 進行互動測試

## 風險 / 取捨

- [風險] 瀏覽器 tab 在背景時，鬧鐘可能延遲或漏響
  → 緩解：使用區間時間判定，並在 `visibilitychange` 回到前景時立即補償檢查。同時引入桌面 Notification。
- [風險] 瀏覽器對自動播放音效有限制（Autoplay Policy）
  → 緩解：提供「測試音效與授權」按鈕引導互動；並在 Modal 播放失敗時提供 UI 提示。
- [風險] 純前端鬧鐘依賴分頁開啟，若關閉分頁或休眠則無法運作
  → 緩解：在 UI 醒目處顯示「免責聲明與使用提示」，提醒使用者保持網頁開啟。
- [風險] localStorage 有容量限制
  → 緩解：限制最大鬧鐘設定數為 100 個，並在儲存時進行 try-catch 容錯處理。
- [取捨] 不支援鬧鐘重複模式（僅單次，觸發後自動轉為 disabled 停用狀態）。
